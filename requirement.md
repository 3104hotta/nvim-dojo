# Neovim 実践学習アプリ — 要件定義

## 概要

プラグインなしの標準 Neovim を、実践課題を通じて体系的に学ぶ学習アプリ。
全9章構成で、各章にミッション形式の演習問題を設ける。

---

## 目的

**「思考の速度で操作できるようになること」** を最終ゴールとする。

nvim の操作が無意識レベルで身につくには、一度学んで終わりではなく **毎日触れ続けること** が必要。そのため本アプリは以下の2軸で設計する。

### 軸1：いつでも見返せる

- どの章・どの演習からでも単独で取り組める構成にする
- 「今日は5分しかない」でも開ける粒度にする（ミッション1本 = 5〜10分が目安）
- README を読むだけでコマンドを思い出せる、簡潔な解説を各章冒頭に置く
- コマンドの意味・理由まで書き、「なぜそう動くのか」が体感できるようにする

### 軸2：毎日続けられる

- 演習の難易度を細かく刻み、詰まって諦めない設計にする
- 1日1ミッションから始められる（完走を強制しない）
- 繰り返し解くことに価値がある題材にする（同じファイルを別の方法で解く、など）
- 上達を実感できるよう、初級→上級の難易度差を明確にする

---

## 対象ユーザー

- vim の基本操作（hjkl・モード切り替え）は知っている
- プラグインに頼らず標準 nvim を使いこなしたい
- 手を動かしながら覚えたいタイプ

| 章 | タイトル | 学習内容 |

## 演習問題の構造

### 各章の構成

```
各章
├── 解説（コマンド・概念の説明）
├── ミッション 1（基本）
├── ミッション 2（応用）
├── ミッション 3（発展）
└── 章末総合演習（その章の学習内容を横断）
```

### 演習問題の形式

各ミッションは以下の要素を持つ。

- **題材ファイル**: 演習に使う実際のコード（Rust / TOML / YAML / Markdown）
- **課題**: 「このファイルに対して、○○の操作を行え」という指示
- **制約**: 「マウス禁止」「特定のコマンドのみ使用」などの縛り
- **ゴール**: 操作後の期待される状態（before/after で明示）
- **ヒント**: 詰まった場合に参照できるヒント（折りたたみ推奨）
- **解答例**: 想定キー操作の手順
- **別解**: より効率的な別のアプローチ（「上達したら試してみよう」として提示）

### 見返し・繰り返しのしやすさ

- 各ミッションの README は **単体で完結** し、前の章を参照しなくても取り組める
- 解答を見た後でも「今度は別の方法で解けるか」を試せるよう、別解を必ず記載する
- 各章冒頭の解説は **コマンド早見表形式** で書き、リファレンスとして機能させる

---

## 章末総合演習

- 各章の末尾に配置
- その章で学んだ全コマンドを組み合わせて解く
- 実際の開発シナリオに近い題材を使用（例：バグ修正、リファクタ、設定変更）
- before/after の状態を明示する

---

## 最終総合演習（全章横断）

全9章の学習内容を統合した総合演習。難易度を3段階に分ける。
**何度でも繰り返し取り組むことを前提とし、毎回タイムを計って自己ベストを更新する楽しみ方もできる設計にする。**

### 初級編

- 対象：各章のミッション 1〜2 レベルをクリアできた人
- 内容：単一ファイルの編集、基本的な移動・編集・検索の組み合わせ
- 目安：1問あたり5〜10分

### 中級編

- 対象：各章の章末総合演習までクリアできた人
- 内容：複数ファイルにまたがる編集、マクロ・vimgrep・バッファ操作の組み合わせ
- 目安：1問あたり15〜30分

### 上級編

- 対象：全章を習得し、実務で nvim を使っている人
- 内容：実際の OSS コードを題材にした大規模リファクタ、LSP + vimdiff + マクロの複合操作
- 目安：1問あたり30〜60分

## 第6章 補足：ウィンドウサイズ変更の演習設計

ウィンドウサイズの変更は「知っている」と「無意識にできる」の差が大きいため、専用のミッションを設ける。

### 習得目標

マウスを一切使わず、思った通りの画面レイアウトをキーボードだけで瞬時に組めること。

### 対象コマンド

| コマンド | 動作 |
|----------|------|
| `Ctrl+w +` / `Ctrl+w -` | 高さを1行増減 |
| `Ctrl+w >` / `Ctrl+w <` | 幅を1列増減 |
| `{N}Ctrl+w +` | 高さを N 行増やす |
| `Ctrl+w =` | 全ウィンドウを均等化 |
| `Ctrl+w _` | 高さを最大化 |
| `Ctrl+w \|` | 幅を最大化 |
| `:resize {N}` | 高さを N 行に指定 |
| `:vertical resize {N}` | 幅を N 列に指定 |

### ミッション構成（第6章内）

**ミッション A — 比率を作る**
- 3ペイン（左:中:右 = 2:3:1 の比率）を指示通りの幅に整える
- `:vertical resize` と `Ctrl+w >/<` の使い分けを体得する

**ミッション B — 素早いフォーカスシフト**
- エディタ + ターミナル のレイアウトで、コードを書いて実行する往復作業を行う
- `Ctrl+w _` でエディタを広げ、ターミナルへ移ったら `Ctrl+w =` で戻す操作を繰り返す

**ミッション C — タイムアタック**
- 指定レイアウト（画像または ASCII アート で提示）を再現するまでの時間を計る
- 繰り返し挑戦して自己ベストを更新することを目的とする
- 別解（最小キーストローク数）を解答例に明示する

**章末総合演習（第6章）**
- Rust プロジェクトを想定した4ペイン構成（`main.rs` / `lib.rs` / `Cargo.toml` / terminal）を組み、指定の編集作業を行う
- ウィンドウの分割・移動・サイズ変更・均等化・最大化をすべて使う流れにする

## 進捗管理

### 方針

- **ファイルベースで完結**させる（外部サービス・DB 不要）
- nvim で開いてそのまま更新できる形式にする
- `progress.md` 1ファイルを見れば全体の進捗が一目でわかるようにする

### progress.md の構造

リポジトリルートに `progress.md` を置く。完了したら `[ ]` を `[x]` に変えるだけ。

```markdown
# 進捗管理

## 第1章 移動の高速化

- [x] ミッション 1（基本）
- [x] ミッション 2（応用）
- [ ] ミッション 3（発展）
- [ ] 章末総合演習

## 第2章 モードとインサート

- [ ] ミッション 1（基本）
- [ ] ミッション 2（応用）
- [ ] ミッション 3（発展）
- [ ] 章末総合演習

...（以下、全章分）

## 最終総合演習

- [ ] 初級編
- [ ] 中級編
- [ ] 上級編
```

## ファイル構成（想定）

- **実際のコードに近い内容**を使う（ダミーの hello world は避ける）
- 言語は **Rust をメイン**とし、TOML（Cargo.toml）・YAML・Markdown・JSON も使用
- 各ミッションの題材は独立していて、前のミッションの結果に依存しない
- before/after は Git の diff 形式または並列表示で明示する

---

## 制約・前提

- プラグインは一切使用しない（標準 nvim のみ）
- nvim バージョン: **0.10 以上**（組み込み LSP を使用するため）
- LSP サーバ（`rust-analyzer`）のインストールは別途必要（演習内で案内する）
- OS: macOS / Linux を想定（Windows は未サポート）
