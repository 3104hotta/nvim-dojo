# 設計メモ — Rust 知識前提の課題（未決定）

## 背景

題材を Rust メインで作ったため、一部のミッションで Vim 操作以外に Rust 知識を要求してしまっている。
本アプリの目的は **Vim を学ぶこと** なので、Rust の理解で詰まるのは設計上の課題。

特に「型を `u32` → `Duration` に変える」のようなカスケード変更は、
Rust に慣れていないと「どこを連鎖して直すべきか」を見つけにくい。

---

## 主な「Rust知識が必要」な箇所

| 章 | ミッション | 問題点 |
|----|----------|--------|
| 1 | mission-2 | `u32` → `Duration` のカスケード（`timeout: 30` → `Duration::from_secs(30)` を全箇所） |
| 1 | summary | `String` → `url::Url` の連鎖変更 |
| 2 | mission-2 | `Vec<u8>` → `&[u8]` で借用ルールが絡む |
| 3 | summary | `unwrap()` → `?` の前提（戻り値型が `Result` であること） |
| 4 | summary | `Box::new(Error::from(...))` → `.into()` の意味 |
| 7 | 全般 | LSP 章なので Rust 知識を多少前提とせざるを得ない |

他にもカスケード型の変更を要求しているミッションが10〜15ある見込み。

---

## 検討した4案

| 案 | 工数 | 説明 | tradeoff |
|----|------|------|----------|
| ① 課題を完全に機械的にする | 中 | 「型を Duration に」ではなく「7行目を xxx に書き換え、12行目を yyy に書き換え」と全変更箇所を列挙 | Vim操作だけに集中できる。READMEが冗長になる |
| ② 連鎖の少ない題材に書き換え | 大 | 該当 exercise.rs を、変数リネーム・コメント追加・行削除など「単純で連鎖しない変更」中心に作り直す | Rust知識不要になる。題材が「実コードっぽさ」を失う恐れ |
| ③ Markdown/TOML/JSON など非 Rust を増やす | 大 | 一部のミッションを非コード題材に切り替え | 学べる「現実味」は減る |
| ④ 各ミッションに「Rustの背景」セクションを追加 | 小 | 「なぜこの変更が必要か」を README で説明 | 変更箇所はRustが分かる人ならわかる、という前提のまま |

---

## 推奨案：① + ④ のハイブリッド

- カスケードする変更には README で **全変更箇所を行番号つきで列挙**（①）
- なぜその変更をするのかを **背景セクションで 1〜2 行説明**（④）
- 題材ファイル自体は変えない（実コードの雰囲気は残す）

### 例（第1章 mission-2 の README イメージ）

```markdown
## 課題

以下の変更を行え（行番号は exercise.rs のもの）:

1. **L1付近**: `use std::io;` の次の行に `use std::time::Duration;` を追加
2. **L19付近**: `timeout: u32,` を `timeout: Duration,` に変更
3. **L31付近**: `timeout: 30,` を `timeout: Duration::from_secs(30),` に変更
4. **L48付近**: `with_timeout(timeout: u32)` を `with_timeout(timeout: Duration)` に変更
5. **L150付近** (`fn cleanup`): `unimplemented!()` を `Ok(())` に変更

## 背景（読み飛ばし可）

`u32`（秒数を表す数値）を `Duration`（Rust 標準の時間型）に置き換える典型的な型安全リファクタ。
型を変えると、その型を使う箇所すべてを揃えて変更する必要がある。
本ミッションでは「ファイル全体に散らばった関連箇所」を見つけて修正する Vim 操作を練習する。
```

### 工数見積

該当ミッションは 10〜15 個。1 ミッションあたり 10 分程度の修正作業。
合計 2〜3 時間程度で完了する見込み。
