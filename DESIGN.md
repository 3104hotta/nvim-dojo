# 設計メモ — Rust 知識前提の課題（未決定）

## 背景

題材を Rust メインで作ったため、一部のミッションで Vim 操作以外に Rust 知識を要求してしまっている。
本アプリの目的は **Vim を学ぶこと** なので、Rust の理解で詰まるのは設計上の課題。

特に「型を `u32` → `Duration` に変える」のようなカスケード変更は、
Rust に慣れていないと「どこを連鎖して直すべきか」を見つけにくい。

---

## 主な「Rust知識が必要」な箇所

| 章 | ミッション | 問題点 |
|----|----------|--------|
| 1 | mission-2 | `u32` → `Duration` のカスケード（`timeout: 30` → `Duration::from_secs(30)` を全箇所） |
| 1 | summary | `String` → `url::Url` の連鎖変更 |
| 2 | mission-2 | `Vec<u8>` → `&[u8]` で借用ルールが絡む |
| 3 | summary | `unwrap()` → `?` の前提（戻り値型が `Result` であること） |
| 4 | summary | `Box::new(Error::from(...))` → `.into()` の意味 |
| 7 | 全般 | LSP 章なので Rust 知識を多少前提とせざるを得ない |

他にもカスケード型の変更を要求しているミッションが10〜15ある見込み。

---

## 検討した4案

| 案 | 工数 | 説明 | tradeoff |
|----|------|------|----------|
| ① 課題を完全に機械的にする | 中 | 「型を Duration に」ではなく「7行目を xxx に書き換え、12行目を yyy に書き換え」と全変更箇所を列挙 | Vim操作だけに集中できる。READMEが冗長になる |
| ② 連鎖の少ない題材に書き換え | 大 | 該当 exercise.rs を、変数リネーム・コメント追加・行削除など「単純で連鎖しない変更」中心に作り直す | Rust知識不要になる。題材が「実コードっぽさ」を失う恐れ |
| ③ Markdown/TOML/JSON など非 Rust を増やす | 大 | 一部のミッションを非コード題材に切り替え | 学べる「現実味」は減る |
| ④ 各ミッションに「Rustの背景」セクションを追加 | 小 | 「なぜこの変更が必要か」を README で説明 | 変更箇所はRustが分かる人ならわかる、という前提のまま |

---

## 採用方針：案② をベース、第7章のみ案④ で補う

**確定事項:**

- **基本は案②**（題材を「連鎖しない単純編集」中心に書き換え）
- **第7章（LSP）など、題材そのものに Rust 文脈が必要なミッションのみ案④**（README に背景説明を追加）で補う
- **案①・案③は不採用**

**理由（Vim 学習が主目的という観点で）:**

- 案①+④ は「行番号で全変更箇所を列挙」になり、README を見ながら機械的に作業する形になりがち。Vim 操作を自分で考える余地が減る。
- 案② なら「どこを直すか」をユーザーが検索・移動コマンドで自分で見つける必要があり、Vim 操作の選択が学習の中心として残る。
- 案② の弱点だった「実コードっぽさが薄れる」は、Vim 学習にとっては実害なし。

### 旧推奨案（参考、不採用）

- カスケードする変更には README で **全変更箇所を行番号つきで列挙**（①）
- なぜその変更をするのかを **背景セクションで 1〜2 行説明**（④）
- 題材ファイル自体は変えない（実コードの雰囲気は残す）

### 例（第1章 mission-2 の README イメージ）

```markdown
## 課題

以下の変更を行え（行番号は exercise.rs のもの）:

1. **L1付近**: `use std::io;` の次の行に `use std::time::Duration;` を追加
2. **L19付近**: `timeout: u32,` を `timeout: Duration,` に変更
3. **L31付近**: `timeout: 30,` を `timeout: Duration::from_secs(30),` に変更
4. **L48付近**: `with_timeout(timeout: u32)` を `with_timeout(timeout: Duration)` に変更
5. **L150付近** (`fn cleanup`): `unimplemented!()` を `Ok(())` に変更

## 背景（読み飛ばし可）

`u32`（秒数を表す数値）を `Duration`（Rust 標準の時間型）に置き換える典型的な型安全リファクタ。
型を変えると、その型を使う箇所すべてを揃えて変更する必要がある。
本ミッションでは「ファイル全体に散らばった関連箇所」を見つけて修正する Vim 操作を練習する。
```

### 工数見積

該当ミッションは 10〜15 個。1 ミッションあたり 10 分程度の修正作業。
合計 2〜3 時間程度で完了する見込み。

## 案2 のイメージ（第1章 mission-2 を例に）

案3（非Rust題材化）は **却下**。案2（連鎖の少ない題材に書き換え）を具体化したもの。

### 現状（カスケード問題あり）

> `timeout` の型を `u32` → `Duration` に変更する

→ 型を変えると `timeout: 30,` も `with_timeout(timeout: u32)` も連鎖修正が必要。
「どこを直すか」を Rust 知識で見つける必要がある。

### 案2 で書き直した README（課題部分）

```markdown
1. ファイル末尾の `fn cleanup` 関数内のコメント
   `// TODO: implement` を `// done` に変更する
2. ファイル先頭の `use std::io;` の次の行に
   `use std::collections::BTreeMap;` を追加する
3. ファイル中央付近の `Default` 実装で
   `"127.0.0.1"` を `"0.0.0.0"` に変更する
4. 同じ `Default` 実装の `port: 8080` を `port: 9090` に変更する
```

### ポイント

- 各編集が **ローカルに閉じている**（型変更のような連鎖を要求しない）
- 「文字列リテラル」「数値」「コメント」「use 追加」だけなので Rust 知識ゼロでも diff が取れる
- ファイルは長いままなので、**スクロール練習**という本来の目的は変わらない
- 編集箇所が「先頭・中央・末尾」に散らばる構造も維持

### トレードオフ

- 良い点: Rust 知識完全に不要・`goal.rs` との diff も明確
- 弱い点: 「実コードのリファクタっぽさ」は薄れ、ややワーク感が出る
  （ただし **Vim 学習が目的** ならこれで十分）

## 演習リセットの運用

### 課題

`exercise.rs` を編集して演習を終えた後、そのまま commit してしまうと
次回演習時には「最初から `goal.rs` と同じ状態」になってしまい、再演習できない。

### 採用方針：案B + 案C のセット

**案B: リセットスクリプト**

- `scripts/reset.sh`（仮）を用意
- 引数なしで全 `exercise.rs` を `HEAD` に戻す
- 引数で章/ミッションを指定したら、その範囲だけリセットも可能
- 実装は `git checkout HEAD -- chapters/**/exercise.rs` 系のコマンドで十分

**案C: README に明記**

- 各ミッション README の末尾に「演習後のリセット」セクションを追加
- 内容: `scripts/reset.sh` の使い方、または個別に `git checkout -- exercise.rs`
- 初見の人が commit/push して詰む事故を防ぐ

### 実装状況

- ✅ `scripts/reset.sh` 実装済み（引数なしで全リセット、章・ミッション単位指定も可）
- ✅ 各ミッション README へのリセット手順追記済み（exercise.* がある 22 ファイル）
